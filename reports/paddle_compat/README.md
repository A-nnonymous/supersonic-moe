## 1. Methodology

| BF16 path | moe_TC_softmax_topk_layer (raw routing) | same |
| FP8 path | moe_TC_softmax_topk_layer + route-level padding | E=8: same; E>8: token rounding |
| Metric | nsys GPU-projection (merged kernel intervals) | same |

> **Caveat**: E>8 FP8 baseline used token rounding (moe_general_routing_inputs),
> while PD uses route-level padding (moe_TC_softmax_topk_layer). The FP8 delta for E>8
> conflates Paddle overhead with routing strategy change. E=8 is strictly apples-to-apples.

## 5. Conclusions

1. **Precision**: 54/54 PASS. All shapes produce valid output under Paddle compat.
2. **BF16 overhead: +2.9% mean** from paddle.enable_compat() proxy dispatch layer.
3. **FP8 overhead: -0.5% mean** (noise floor). CUTLASS kernels bypass the proxy.
4. **E=8 FP8 (strict fair): -0.6% mean** confirms zero Paddle overhead on FP8.
5. **FP8 speedup**: PD 1.27-1.79x vs PT 1.29-1.70x (27 shapes).
6. **Memory delta** from API level (full module vs bare function), not compat layer.
